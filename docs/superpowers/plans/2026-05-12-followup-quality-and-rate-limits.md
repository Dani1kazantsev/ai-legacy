## Проблема 1: Rate limits на Groq free tier

### Симптомы
1. **413 Payload Too Large** — после 3-4 сообщений промпт перевалил за 12000 токенов (лимит llama-3.3-70b на free tier).
2. **429 Too Many Requests** — между сообщениями упирались в rate limit по запросам в минуту.

### Почему так
- System prompt сам по себе ~11600 токенов (поведенческий профиль, principles, opinions, humor, relationships, phrases, RAG, few-shot).
- После 3-4 сообщений история диалога добавляет ~1000-1500 токенов → превышаем 12000.
- На каждое сообщение бот пересобирает промпт + ходит в RAG → лимит TPM (tokens per minute) выгорает.

### Проверенные модели на Groq
| Модель | TPM лимит | Контекст | Заметка |
|---|---|---|---|
| llama-3.3-70b-versatile | **12k** | 131k | Текущая, упираемся |
| llama-3.1-8b-instant | ~6k | 131k | Хуже качество и хуже лимит |
| qwen/qwen3-32b | **6k** | 131k | Жёсткий лимит, не вариант |
| meta-llama/llama-4-scout-17b | работает | 131k | Не проверял лимит точно |
| openai/gpt-oss-20b | работает | 131k | Не проверял |
| openai/gpt-oss-120b | работает | 131k | Не проверял |
| gemma2-9b-it | **decommissioned** | — | Убрана из Groq |

### Возможные пути решения

#### А. Платный Groq Dev Tier (~$25/мес) — самое простое
- Снимает rate limits полностью
- Качество llama-3.3-70b останется
- Проблема: иностранная карта нужна (у пользователя её нет)
- ❌ Не подходит из-за оплаты

#### Б. Российский LLM-аггрегатор (vsegpt.ru, proxyapi.ru, neuroapi.host)
- Оплата рублями, СБП
- OpenAI-совместимый API → меняется только `LLM_BASE_URL` и ключ
- Доступ к Claude, GPT, Gemini, Llama — выбор лучшего качества
- Цены ~10-30% наценки над прямым API
- ✅ Самый практичный путь

#### В. Сократить system prompt
- Сейчас 11600 токенов — это много
- Можно сократить relationships.md (10422 chars) и opinions.md (7686 chars) — самые жирные
- Минус: теряем смысл, который пользователь специально наговорил в интервью
- Альтернатива: разделить personality на "always-in" (~5k токенов) и "on-demand RAG" (остальное)

#### Г. Попробовать другие модели Groq (gpt-oss-120b, llama-4-scout)
- Лимиты не проверены, могут быть таким же боль
- Качество для русского — неизвестно
- Худший случай: качество ниже llama-3.3-70b

#### Д. Architecture: tiered prompt
- Stage 1: лёгкий промпт с behavior.md + few-shot для коротких сообщений
- Stage 2: полный промпт с RAG только если сообщение содержит триггеры (имена близких, темы)
- Сложнее в реализации, но решает обе проблемы

#### Е. Architecture: prompt caching
- Если LLM-провайдер поддерживает кэширование статичной части промпта
- Groq не поддерживает; Anthropic поддерживает; OpenAI имеет automatic caching
- Решает rate limit проблему частично

## Проблема 2: Качество ответов — «тупо рандом фраз»

### Симптомы
Из реального теста:
- Пользователь: «Дарова»
- Бот: «Ахаха, что там?» — звучит ок
- Пользователь: «Где?»
- Бот: «Ща найти! 😁» — **бред** (откуда «найти»?)
- Пользователь: «Что найти, про что ты вообще?»
- Бот: фейл (413)

Пользователь сформулировал: **«тупо рандом фраз, это что за фигня»**.

### Гипотезы

1. **Модель цепляется за случайные few-shot примеры.** При sample_examples(n=30) бот видит случайные пары вход-выход из всех 4887 сообщений. На реплику «Где?» он может «вспомнить» пример где кто-то говорил «Где?» и ответ был «Ща найти!» — и тупо повторить, не понимая контекста.

2. **RAG-retrieval отвлекает.** BM25 находит топ-8 по словам в запросе. На «Где?» — это очень общее слово, retrieve может вытащить нерелевантные диалоги.

3. **System prompt перегружен.** 11600 токенов инструкций — модель устает следить за всеми правилами и начинает галлюцинировать.

4. **Llama 3.3 70B не идеально следует длинным русским инструкциям.** Возможно, более качественная модель (Claude, GPT-4) даст лучший результат на том же промпте.

5. **Нет адаптации под собеседника.** Бот не знает, кто пишет, поэтому смешивает регистры (стиль из чата с мамой + стиль из чата с братом).

### Возможные пути решения

#### А. Адаптивный few-shot и RAG
- Когда бот понимает, кто пишет (по Telegram ID), подмешивает примеры **только из чата с этим человеком**
- Это уже частично есть в дизайне (Plan 2 раздел "Адаптивный few-shot"), но не реализовано
- Решает проблему 5

#### Б. Качественная модель
- Попробовать через vsegpt.ru → Claude Sonnet 4.6
- Claude лучше следует инструкциям и держит регистр
- Решает проблемы 3 и 4

#### В. Чистить RAG-retrievals по релевантности
- Сейчас возвращаем топ-8 по BM25 score, но не фильтруем по «достаточно ли релевантно»
- Установить минимальный порог score
- Если ничего не нашлось — не подмешивать RAG-блок вообще
- Решает проблему 2

#### Г. Уменьшить few-shot
- 30 случайных примеров — это много шума
- Уменьшить до 10-15
- Или сделать примеры **тематически отобранными** (не случайными)

#### Д. Структурировать system prompt лучше
- Сейчас всё в одном плоском блоке
- Можно добавить **примеры неправильных ответов** и явно сказать «не делай так»
- Можно добавить **пошаговую инструкцию** (think then respond)

#### Е. Калибровка через user feedback
- Команда `/forget` уже задумана, но не реализована
- Добавить команды: «нет, я бы так не сказал» (исправление), «продолжай в том же духе» (валидация)
- Использовать эти сигналы для обновления промпта или few-shot выборки

## Рекомендованный план действий для следующей сессии

### Шаг 1: Переход на качественный LLM через российский аггрегатор
1. Зарегиться на **vsegpt.ru** или **proxyapi.ru** (5-10 мин, рублёвая оплата)
2. Пополнить на 500р для тестов
3. Получить API ключ для **Claude Sonnet 4.6** или **GPT-4** через их OpenAI-совместимый endpoint
4. Обновить `.env`:
   ```
   LLM_API_KEY=<новый ключ>
   LLM_MODEL=<claude-sonnet-4-6 или gpt-4o>
   LLM_BASE_URL=<endpoint vsegpt>
   ```
5. Перезапустить бот, проверить что качество выше

### Шаг 2: Если качество всё равно «рандом» — диагностика prompt'а
1. Залогировать **полный system prompt** в лог-файл при каждом запросе
2. Проверить **какие именно** RAG-примеры подтягиваются на «Где?» и «Что найти»
3. Если RAG-выборка плохая → поднять порог релевантности или отключить для коротких реплик
4. Если few-shot сбивает с толку → уменьшить до 5-10 или сделать постоянными (одни и те же лучшие примеры)

### Шаг 3: Адаптивный few-shot per-recipient
Сейчас в single-user mode это не критично, но **для Plan 4** (дерево доверия) обязательно:
- Маппинг telegram_user_id → имя в `relationships.md`
- На каждом запросе: сначала берём примеры **из чата с этим человеком**, потом добивает рандомом
- Обновить `few_shot.py` и `handler.py`

### Шаг 4: Retry/backoff на 429
Добавить в `src/ai_legacy/llm.py` exponential backoff при rate limit ошибке.

### Шаг 5: Token budget tracking
Перед отправкой считать tokens, и если превышаем — обрезать историю или RAG-блок.
